# Performance baseline ‚Äî 2026-05-28

Captured against a **local production build** (`pnpm build && pnpm start`) wired
to the production Neon Postgres + Google Drive storage. Phase 0 Server-Timing
instrumentation is on this build, so `/api/files` per-stage timings are real.

> **Caveats vs Vercel-sin1 numbers** ‚Äî actual production will be **faster** for
> the `storage` segment because Drive RTTs from `sin1` are shorter than from
> the laptop this was captured on (India, residential ISP).  DB segment is
> roughly representative because the DB is the same Neon endpoint.  Phase 1
> will re-capture against production after the rebuild deploys.
>
> **Why not against the Vercel preview?** Three sequential preview builds
> served `/api/files` responses **without** a `server-timing` header despite
> the route handler clearly setting it ‚Äî most likely a Vercel build-artifact
> cache held the pre-Phase-0 compilation.  Local prod-build was the
> deterministic path to actual numbers.

## `/api/files` Server-Timing (median of 8 authenticated requests against `/patients`)

| Stage                              | Median (ms) |
|------------------------------------|------------:|
| `auth` (NextAuth + DB user lookup) | **134**     |
| `db` mark (cumulative through `getMediaForRead`)  | **247**  |
| `storage` mark (cumulative through Drive `get`)   | **2,174** |
| `total`                            | **2,175**   |

**Per-stage time (subtracting from previous mark):**

| Stage                                                            | Median (ms) | % of total |
|------------------------------------------------------------------|------------:|-----------:|
| `auth` alone (`getCurrentUser()` ‚Äî DB user lookup)               | **134**     | 6.2% |
| `getMediaForRead` (1 asset query + up to 3 parallel link checks) | **113**     | 5.2% |
| Storage / Drive (`files.get` metadata RTT + stream RTT)          | **1,927**   | **88.6%** |
| **Total**                                                        | **2,175**   | 100% |

**Drive is the entire performance story.** Phase 1's signed-URL path drops
both `auth` and `getMediaForRead` chains (no cookie auth, single asset
lookup) and Phase 1's `getStreamOnly` drops the Drive metadata RTT ‚Äî leaving
just one Drive stream call, edge-cached after the first hit.

| Fluid Provisioned Memory          | 9.1 / 360 GB-Hrs |

| ISR Reads                         | 1.7K / 1M        |
| Edge Request CPU Duration         | 6s / 1h          |

**The killer datapoint:** Function Invocations (63K) > Edge Requests (48K).
That inversion means our functions are doing what edge caching should be
doing ‚Äî every `/api/files/*` hit re-runs the auth + DB + 2√ó Drive chain
because the response carries `cache-control: no-cache`.  Phase 1 fixes that.

## How these numbers were captured

1. `git checkout perf-rebuild-plan` (commit `4163850`, includes Phase 0 instrumentation).
2. `pnpm build && pnpm exec dotenv -e .env.local -- env AUTH_URL=http://localhost:3000 pnpm exec next start --port 3000`.
3. `tmp/capture-baseline.mjs` (Playwright Pixel 7 emulation) signs in as the
   test user, navigates to `/patients`, captures every `/api/files/*` response
   header, prints them as JSON.
4. `npx -y lighthouse@12 http://localhost:3000/<path> --form-factor=mobile
   --throttling-method=simulate --screen-emulation.mobile=true
   --extra-headers='{"Cookie":"<session>"}'` for each of the three pages above.
5. Vercel dashboard snapshot read directly from the project's Usage tile.

`tmp/capture-baseline.mjs`, `tmp/lh-*.json`, and `tmp/cookie.txt` are scratch
artifacts in `tmp/` which is git-ignored ‚Äî they are NOT committed.
